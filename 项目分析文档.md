## 一、项目概述

游领资本（品牌名 GHOST）是一个面向国内 Web3 社群的活动汇总与用户管理前端应用。核心功能包括：空投活动发布与参与、新手教育、合约教学、实时K线看盘工具、用户资料管理、积分兑换系统，以及开发中的运营端内容管理。

**定位**：运营方统一发布活动/空投信息，引导新手学习，管理用户资料与积分，为社群成员提供清晰的参与路径。

## 四、路由结构（状态驱动）

项目**没有使用 react-router**，在 `App.js` 中通过 `currentSection` 状态控制当前显示的组件。刷新页面会回到默认第一个 Tab。

### 4.1 认证流程路由

| page 状态 | 显示组件 | 说明 |
|---|---|---|
| `login` | Login | 登录页（默认） |
| `register` | Register | 注册页 |
| `forgot` | ForgotPassword | 找回密码页 |

### 4.2 主页面路由（sections 数组）

| key | 导航标签 | 组件 | 权限要求 |
|---|---|---|---|
| `airdrop` | 空投活动 | Airdrop | is_verified = true |
| `newbie` | 新手知识 | NewbieGuide | is_verified = true |
| `contract` | 合约教学 | ContractTutorial | is_verified = true 且 高级用户/合作机构/premium |
| `tools` | 看盘工具 | TradingTools → BinanceChart | is_verified = true 且 高级用户/合作机构/premium |
| `profile` | 个人中心 | UserProfile | 登录即可 |
| `points` | 积分兑换 | PointsExchange | is_verified = true |

**未上线**：`ContentManagement`（内容管理）已导入但未注册到导航。

### 4.3 权限分级体系

```
未验证用户 ──→ 所有功能模块均不可访问
    │
    ▼ (is_verified = true)
基础用户 ──→ 空投活动 + 新手知识 + 积分兑换 + 个人中心
    │
    ▼ (user_type ∈ {高级用户, 合作机构, premium})
高级用户 ──→ 基础功能 + 合约教学 + 看盘工具
```

## 五、各页面功能详解

### 5.1 认证系统

#### AuthContainer（认证容器）
- 管理登录/注册/找回密码三个子页面之间的切换
- 动态加载 particles.js 粒子特效背景（金色粒子网络）
- 粒子参数：60 个粒子，金色 `#bfa14a`，支持鼠标排斥和点击添加粒子

#### Login（登录页）
- **表单字段**：邮箱、密码
- **校验**：邮箱格式实时校验（onBlur 触发）
- **登录流程**：
  1. Supabase `users` 表按 username（实际存邮箱）查询
  2. `bcrypt.compare` 比对密码哈希
  3. 调用 `useUserStore.login()` 写入全局状态
- **查询字段**：id, username, password_hash, nickname, points, invite_code, parent_invite_code, binance_uid, bg_uid, created_at

#### Register（注册页）
- **表单字段**：邮箱、密码、确认密码、邀请码（可选）
- **校验规则**：邮箱格式、密码最少 8 位、两次密码一致
- **注册流程**：
  1. 查重（同一 username 不可重复）
  2. `bcrypt.hash` 加密密码（saltRounds=10）
  3. 生成 6 位随机邀请码（字符集 `ABCDEFGHJKLMNPQRSTUVWXYZ23456789`，排除易混淆字符）
  4. 插入 Supabase `users` 表
- **提示方式**：SweetAlert2 弹窗

#### ForgotPassword（找回密码）⚠️ 未完成
- **三步流程设计**：发送验证码 → 验证验证码 → 设置新密码
- **当前实现**：验证码固定为 "123456"，密码重置仅弹提示，**未实际写入数据库**

### 5.2 空投活动（Airdrop）

**数据来源**：Supabase `airdrops` 表

**核心功能**：
- 活动列表展示：按状态优先级排序（即将开始 > 进行中 > 已结束）
- 活动状态标签：`upcoming`（金色）、`ongoing`（绿色）、`ended`（红色）
- 活动详情弹窗：通过 `marked` 库将 Markdown 渲染为 HTML
- 活动参与提交

**活动卡片信息**：活动名称、状态标签、奖励描述、截止时间、开始时间

**参与流程**：
1. 点击"查看活动详情"→ SweetAlert2 弹窗展示 Markdown 内容
2. 仅 `ongoing` 状态活动可参与
3. 点击"立即参与"→ 弹出提交表单（文字描述 + 图片上传）
4. 图片上传至 Supabase Storage `activity_images` bucket
5. 插入记录到 `activity_submissions` 表
6. 调用 RPC `increment_participants` 增加参与人数

### 5.5 看盘工具（TradingTools + BinanceChart）

#### TradingTools（交易对选择器）
- **交易对**：BTC/USDT, ETH/USDT, BNB/USDT, ADA/USDT
- **时间周期**：1m, 5m, 15m, 1H, 4H, 1D
- **技术指标按钮**：MA, RSI, MACD, BB, KDJ（UI 存在，实际未接入图表）

#### BinanceChart（核心图表组件）

- **RSI 计算**：前端自行计算 14 周期 RSI，绘制超买(70)/超卖(30)参考线

- **时间轴同步**：K线、交易量、RSI 三图时间轴联动

## 六、状态管理（Zustand）

### 6.1 Store 结构

文件：`src/store/userStore.js`

```javascript
{
  isLoggedIn: false,
  userInfo: {
    id: null,                  // 用户 ID (Supabase 主键)
    username: '',              // 显示名称 (优先取 nickname)
    email: '',                 // 邮箱 (存在 username 字段)
    avatar: '',                // 头像 emoji
    joinDate: '',              // 注册日期
    level: '普通用户',          // 用户等级
    points: 0,                 // 积分
    parentInviteCode: '',      // 父邀请码
    myInviteCode: '',          // 自己的邀请码
    binanceUID: '',            // 币安 UID
    bitgetUID: '',             // Bitget UID
    wechat: '',                // 微信号
    okxUID: '',                // OKX UID
    is_verified: false,        // 认证状态
    user_type: '普通用户',      // 用户类型
    premium_days: 0,           // 高级会员剩余天数
  }
}
```

### 6.2 Store 方法

| 方法 | 功能 |
|---|---|
| `login(userData)` | 登录，调用 `fetchUserByUsername` 从 Supabase 获取完整用户信息并写入 store |
| `logout()` | 登出，重置所有状态到默认值 |
| `updateUserInfo(newInfo)` | 合并更新用户信息 |
| `fetchUserByUsername(username)` | 从 Supabase `users` 表查询并更新 store |
| `updatePoints(points)` | 增量更新积分 |
| `getUserInfo()` | 获取当前用户信息快照 |
| `getLoginStatus()` | 获取登录状态 |

### 6.3 数据流

```
用户登录 → Login 组件 bcrypt.compare 验证密码
         → store.login() → fetchUserByUsername()
         → Supabase SELECT users → 映射字段 → persist 到 localStorage

组件读取 → const { userInfo } = useUserStore() → 响应式更新

信息更新 → store.updateUserInfo() → 合并 → persist 自动同步 localStorage
```

### 6.4 持久化与迁移
- persist 中间件将 store 持久化到 localStorage（key: `user-storage`）
- `version: 1` + `migrate` 函数确保旧版本数据迁移时 `is_verified` 和 `user_type` 有默认值

### 11.2 能否支撑 500-1000 人使用？

**结论：当前架构在现有基础设施下可以支撑 500-1000 人同时在线，但存在明显的瓶颈和风险。**

#### 能撑住的部分

| 方面 | 分析 |
|---|---|
| Supabase 免费版 | 支持 500 并发连接、500MB 数据库、1GB 文件存储，500 人轻量使用可覆盖 |
| 币安 K 线 WebSocket | 每个 BinanceChart 建立独立 WebSocket 连接，币安限单 IP 5 条/秒，10 人同时看盘无压力 |
| 状态管理 | Zustand 轻量，localStorage 持久化不依赖服务端，前端本身无压力 |
| 静态页面渲染 | CRA 纯 SPA，CDN 分发即可，页面渲染无瓶颈 |

#### 瓶颈和风险

| 风险 | 严重程度 | 说明 |
|---|---|---|
| **无后端中间层** | 🔴 严重 | 前端直连 Supabase，anon key 暴露意味着任何用户可以用该 key 操作数据库（读写任意表），无服务端权限校验 |
| **Supabase Row Level Security (RLS)** | 🔴 严重 | 代码中未见 RLS 策略配置。如果 Supabase 未配置 RLS，anon key 可以查询所有用户数据（包括 password_hash） |
| **前端密码哈希** | 🔴 严重 | bcrypt 在前端执行，密码以明文传输到前端后才哈希。应该在服务端哈希 |
| **并发写入** | 🟡 中等 | 积分兑换、活动参与等写入操作无乐观锁/悲观锁，多人同时兑换同一商品可能出现超卖 |
| **实时 K 线连接数** | 🟡 中等 | 每个用户一个 WebSocket 连接。1000 人同时看盘 = 1000 连接，币安可能限流。且全部从用户浏览器直连币安，无服务端缓存 |
| **图片上传** | 🟡 中等 | 活动图片直传 Supabase Storage，免费版 1GB 存储上限，500 人上传图片很快用尽 |
| **无分页** | 🟡 中等 | 所有列表（空投、指南、教程、商品）都是 `SELECT *`，数据量增长后前端渲染压力增大 |
| **无路由** | 🟡 中等 | 刷新页面回到第一个 Tab，无 URL 分享能力，用户无法直达特定页面 |
| **localStorage 限制** | 🟢 轻微 | Store 持久化到 localStorage（约 5MB 上限），当前数据量不会触发 |

#### 容量预估

| 用户规模 | Supabase 免费版 | 币安 API | 前端性能 |
|---|---|---|---|
| 100 人 | ✅ 无压力 | ✅ 无压力 | ✅ 无压力 |
| 500 人 | ⚠️ 接近连接上限和存储上限 | ✅ 看盘用户 < 100 人时无压力 | ✅ 无压力 |
| 1000 人 | ❌ 需要升级 Supabase Pro 版 | ⚠️ 同时看盘用户 > 100 时可能限流 | ⚠️ 列表数据量大时需分页 |

### 11.3 核心问题清单

#### 安全问题（必须修复）

1. **前端直连数据库**：需要引入后端中间层（Node.js/Express 或 Supabase Edge Functions），所有数据库操作通过后端 API 代理
2. **密码哈希在前端**：bcrypt 应在服务端执行，前端只传明文密码（通过 HTTPS）
3. **Supabase anon key 暴露**：必须配置 RLS 策略，限制 anon 角色的表级和行级访问权限
4. **无输入过滤**：用户提交的内容（活动描述、个人资料）无 XSS 过滤
5. **推广链接硬编码**：交易所注册推广链接和邀请码直接写在前端代码中

#### 架构问题（建议改进）

1. **引入路由库**：添加 react-router-dom，支持 URL 导航和页面刷新不丢失状态
2. **添加后端中间层**：即使是 Supabase Edge Functions，也需要将敏感操作移到服务端
3. **数据分页**：所有列表接口添加分页参数
4. **代码分割**：使用 React.lazy + Suspense 按路由懒加载组件
5. **样式模块化**：2373 行的 Components.css 拆分为 CSS Modules 或 styled-components
6. **TypeScript 迁移**：提升代码可维护性和类型安全

#### 功能缺失

1. 找回密码功能未完成（模拟验证码 + 未写入数据库）
2. 内容管理功能未上线
3. 赚取积分模块仅有占位内容
4. BtcChart 备用组件为空壳
5. 技术指标按钮（MA/MACD/BB/KDJ）仅 UI，未接入图表逻辑

## 十三、总结

游领资本前端项目目前处于 **MVP/原型阶段**，核心功能（空投活动、新手教育、看盘工具、积分系统）已经可以跑通基本流程，视觉设计统一且有辨识度。但从工程角度看，存在严重的安全架构缺陷（前端直连数据库、前端密码哈希、无 RLS），以及多项未完成的功能模块。

**500 人规模**：在配置好 Supabase RLS 的前提下，免费版可勉强支撑，但需注意存储和连接数上限。

**1000 人规模**：需要升级 Supabase Pro 版、引入后端中间层、添加数据分页，并考虑币安 API 的连接数管理。

**建议**：在正式面向用户前，至少完成 Phase 1（安全加固）和 Phase 2（架构升级）的改造。
